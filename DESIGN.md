# DESIGN.md — Store Intelligence API Architecture

## System Overview

The system converts raw CCTV footage from Apex Retail stores into a queryable analytics API. The pipeline has four stages, each independently testable:

```
Raw CCTV Clips
      │
      ▼
┌─────────────────┐
│ Detection Layer │  YOLOv8n + ByteTracker + Zone Classifier
│  (pipeline/)    │  → Structured JSONL events
└────────┬────────┘
         │  POST /events/ingest (batch, idempotent)
         ▼
┌─────────────────┐
│ Intelligence API│  FastAPI + SQLite (WAL mode)
│   (app/)        │  → /metrics, /funnel, /heatmap, /anomalies, /health
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Live Dashboard  │  Vanilla JS + nginx
│  (dashboard/)   │  → Real-time metrics, heatmap, anomalies
└─────────────────┘
```

## Stage 1 — Detection Layer

### Model Choice: YOLOv8n

YOLOv8 nano was selected for person detection. See CHOICES.md for the full trade-off analysis. Key reasons: strong COCO person-class accuracy, well-maintained Ultralytics ecosystem, runs at acceptable speed on CPU for 1080p@15fps with frame-skipping (every 3rd frame).

### Tracking: ByteTrack-inspired IoU tracker

`pipeline/tracker.py` implements a two-pass Hungarian matching approach:

1. **Pass 1**: High-confidence detections (conf ≥ 0.5) matched to active tracks via IoU cost matrix.
2. **Pass 2**: Unmatched tracks matched against low-confidence detections at a lower IoU threshold.
3. **New tracks**: Remaining unmatched high-confidence detections create new tracks.

Tracks not matched for `MAX_LOST_FRAMES` (30 frames = 2 seconds at 15fps) are moved to `lost_tracks` and emit EXIT events. This 2-second grace period handles momentary occlusion without premature exits.

### Re-ID: Trajectory-based appearance features

Rather than a full torchreid OSNet model (which requires GPU and a fine-tuned retail dataset), Re-ID uses a lightweight 5-dimensional feature vector: `[centroid_x, centroid_y, aspect_ratio, width, height]`.

A rolling mean of the last 50 frames of features per track is compared via cosine similarity against recently-lost tracks when creating a new track. If similarity ≥ 0.72 and the track is in `lost_tracks`, the visitor_id is reused and a `REENTRY` event is emitted instead of `ENTRY`.

**Trade-off**: This approach is sensitive to camera-angle changes and time gaps. For production, replacing with OSNet embeddings from torchreid would significantly improve Re-ID accuracy across longer gaps and partial occlusion.

### Staff Detection

Staff are detected using a torso colour heuristic: extract the middle third of the bounding box height (torso region), convert to HSV, and check for low saturation (grey/black/white uniforms typical in Indian retail). Staff events are still emitted but flagged `is_staff=true` so the API excludes them from customer metrics.

**Limitation**: This heuristic is brittle — a customer in a plain white shirt may be misclassified. For production: train a binary classifier on `[person crop → uniform/non-uniform]` or use a VLM prompt (see AI-Assisted Decisions below).

### Zone Classification

Zone assignment uses ray-casting polygon containment tests against the zone polygons defined in `store_layout.json`. The centroid of each tracked bounding box is tested against each zone polygon. Zones are camera-specific: the floor camera covers product zones, the billing camera covers the billing area.

### Edge Cases

| Edge Case | Handling |
|-----------|----------|
| Group entry | YOLO detects individuals — 3 bounding boxes → 3 tracks → 3 ENTRY events |
| Staff movement | Torso-colour heuristic flags `is_staff=True`; API excludes from metrics |
| Re-entry | ByteTracker Re-ID compares cosine similarity against lost tracks |
| Partial occlusion | Low-confidence events emitted with actual confidence, never dropped |
| Billing queue buildup | `queue_members` set tracks who joined; ABANDON emitted on departure |
| Empty periods | Empty frame arrays → no events emitted; API returns 0, not null |
| Camera overlap | `is_cross_cam_duplicate()` suppresses double-counts in ±10s window |

## Stage 2 — Event Stream

Events are buffered in `EventEmitter` and written to a JSONL file in batches (default: 50). The emitter optionally POSTs batches directly to `POST /events/ingest` during live processing, enabling real-time dashboard updates.

### Schema Decisions

- `event_id`: UUIDv4, generated by the pipeline (globally unique).
- `timestamp`: ISO-8601 UTC, derived from clip start time + frame offset (not wall clock).
- `dwell_ms`: 0 for instantaneous events (ENTRY/EXIT/ZONE_ENTER); populated for ZONE_DWELL.
- `confidence`: Raw model confidence, never suppressed. Low-confidence events are still emitted so the API layer can decide how to handle them.
- `is_staff`: Boolean from detection layer. Staff events reach the API but are filtered out of customer metrics.

## Stage 3 — Intelligence API

### Storage: SQLite with WAL Mode

WAL (Write-Ahead Logging) allows concurrent readers while a single writer is active. This is sufficient for the 40-store scale where events arrive in batches and reads are infrequent. See CHOICES.md for the SQLite vs PostgreSQL trade-off.

### Ingest: Idempotent + Partial Success

`POST /events/ingest` checks `event_id` against the events table before inserting. Duplicate `event_id` → status="duplicate" without error, making the endpoint safe to call twice with the same payload. Malformed events (failed Pydantic validation) return status="error" while valid events in the same batch still proceed.

### Session Management

The `sessions` table tracks visitor sessions, updated on each ENTRY/EXIT/ZONE_ENTER/REENTRY event:
- ENTRY → INSERT session row
- ZONE_ENTER in billing zone → UPDATE `billing_zone_ts`; run POS correlation
- EXIT → UPDATE `exit_ts`
- REENTRY → INSERT new session row (same visitor_id, new entry_ts)

Funnel and metrics queries operate on the `sessions` table (session-level) not raw events. This ensures re-entries don't inflate visitor counts.

### POS Correlation

Conversion detection: if a POS transaction exists within 5 minutes after a visitor was last seen in the billing zone for the same store, the session is marked `converted=1`. This is time-window + store correlation (no customer_id in POS data).

### Anomaly Detection

Anomalies are computed in real-time on each `GET /stores/{id}/anomalies` request. No pre-computed baselines are required for QUEUE_SPIKE (absolute threshold) or DEAD_ZONE (30-min rolling window). CONVERSION_DROP computes a rolling 7-day average on demand.

---

## Stage 4 — Live Dashboard

The dashboard (`dashboard/index.html`) is a single-page app served by nginx. It polls the API every 10 seconds and renders:
- KPI cards (visitors, conversion, dwell, queue, abandonment, anomaly count)
- Conversion funnel bar chart
- Zone heatmap (visit frequency, normalised 0–100)
- Active anomalies list with severity and suggested actions

The dashboard updates in real time as the detection pipeline pushes events to the API — proving the pipeline and API are genuinely connected.

## Production Readiness

### Structured Logging

Every request emits a single JSON log line (via `StructuredLoggingMiddleware`):
```json
{"trace_id": "...", "method": "POST", "path": "/events/ingest",
 "status_code": 207, "latency_ms": 4.2, "store_id": null, "event_count": 50}
```

### Graceful Degradation

- Database unavailable → HTTP 503 with structured body (no raw stack traces)
- `GET /health` continues to function even when the DB is partially unavailable
- Empty store periods return zeros, not null or 500 errors

### Test Coverage

Tests cover:
- All 8 event types (schema compliance, UUID uniqueness, timestamp format)
- Idempotent ingest (duplicate detection)
- Partial success on malformed batches
- Staff exclusion from metrics
- Zero-purchase stores (conversion_rate = 0.0, not null)
- Funnel session deduplication (re-entries don't inflate counts)
- All 5 anomaly types (trigger and non-trigger conditions)
- STALE_FEED detection
- Empty store periods (no crash, no null)

## AI-Assisted Decisions

### 1. ByteTracker vs DeepSORT for Multi-Object Tracking

I used an LLM to compare ByteTrack, DeepSORT, and StrongSORT for this use case. The AI suggested ByteTrack due to its better performance at high occlusion ratios and lack of dependency on a pre-trained ReID model for the association step (it uses IoU only, with ReID as an optional second pass).

**My evaluation**: I agreed with this recommendation. In retail CCTV, the camera is static and occlusion is frequent (people passing in front of each other). ByteTrack's two-pass approach handles this better than DeepSORT's single-pass Kalman filter + appearance distance combination. I implemented a simplified version without the Kalman filter (using raw IoU instead) because the static camera assumption makes velocity prediction less critical.

**Override**: The AI suggested using the full ByteTrack implementation from the `supervision` library. I chose to implement the core matching logic manually in `tracker.py` instead — this gives me more control over the Re-ID integration and avoids a heavy dependency. The trade-off is more code to maintain, but better testability.

### 2. Staff Detection: VLM vs Heuristic

I consulted an LLM about whether to use a VLM (GPT-4V or Claude Vision) for staff detection. The AI's suggestion:

> "Use a VLM with a prompt like: 'Is this person wearing a retail staff uniform? The uniform is [describe colour]. Answer: yes/no with confidence 0-1.' This would give higher accuracy than colour-only heuristics, especially for partial visibility."

**My evaluation**: The VLM approach would be more accurate and would generalise across different store uniform colours. However, it has two problems for this use case:
1. **Latency**: A VLM API call per detection at 5 effective fps would create a bottleneck and incur significant API costs.
2. **Privacy**: Sending cropped person images to an external API raises GDPR/privacy concerns for CCTV footage.

**My decision**: I implemented the HSV torso-colour heuristic for the baseline and documented the VLM approach as the upgrade path. The heuristic is fast and free, but will have false positives on similarly-coloured customer clothing.

### 3. SQLite WAL vs PostgreSQL + TimescaleDB

The AI strongly recommended PostgreSQL with TimescaleDB for a time-series event store at scale. This is the right long-term choice for 40 live stores.

**My evaluation**: For this challenge's scope (single-node, batch + occasional live replay), SQLite with WAL mode is sufficient and eliminates one infrastructure dependency. `docker compose up` starts one container instead of three.

**My override**: I chose SQLite for simplicity of the submission, but explicitly designed the `database.py` `get_connection()` context manager as an abstraction boundary — replacing it with asyncpg would require changes only in that file, not across the codebase.
